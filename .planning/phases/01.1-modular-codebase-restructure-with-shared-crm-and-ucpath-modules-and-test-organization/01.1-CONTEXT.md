# Phase 01.1: Modular Codebase Restructure - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Reorganize the flat `src/onboarding/` structure into shared `crm/` and `ucpath/` modules with workflow-specific `workflows/onboarding/`, and move co-located tests to a `tests/` hierarchy. No new functionality — pure restructure for multi-workflow extensibility.

</domain>

<decisions>
## Implementation Decisions

### Directory structure
- `src/auth/` — stays as-is (shared infrastructure)
- `src/browser/` — stays as-is (shared infrastructure)
- `src/crm/` — generic ACT CRM page interaction (search, navigate, extract fields). Any workflow that needs CRM data reuses this.
- `src/ucpath/` — generic UCPath/PeopleSoft interaction (navigate menus, fill forms). Any workflow that enters data reuses this.
- `src/utils/` — stays as-is (env, log)
- `src/workflows/onboarding/` — onboarding-specific: schema (Zod), extract (FIELD_MAP + field mapping), enter (UC_FULL_HIRE), index (orchestration)
- `src/cli.ts` — routes to workflows
- `tests/unit/` — schema, env, log tests
- `tests/e2e/` — auth, crm, workflow integration tests
- Delete `src/onboarding/` and `src/workflow/` after migration

### CRM extraction boundary
- `extractField(page, label)` moves as-is to `crm/extract.ts` with all 3 strategies (label-based, ARIA, table cell)
- `FIELD_MAP` and `extractRawFields(page)` move to `workflows/onboarding/extract.ts` — onboarding-specific field knowledge
- `workflows/onboarding/extract.ts` imports `extractField` from `crm/extract.ts`

### CRM module scope
- `crm/search.ts` — generic: `searchByEmail(page, email)`, `selectLatestResult(page)`
- `crm/navigate.ts` — generic: `navigateToSection(page, sectionName)` instead of hardcoded `navigateToEntrySheet`. Workflows pass their section name (e.g., "UCPath Entry Sheet")
- `crm/types.ts` — owns `ExtractionError` (CRM-level concern, reused by all workflows)

### Validation ownership
- Zod validation is workflow-specific, not shared
- `crm/` extracts raw strings; each workflow defines its own Zod schema
- `workflows/onboarding/schema.ts` keeps the onboarding EmployeeData schema

### Claude's Discretion
- UCPath module scope — whether to create stubs for `navigate.ts`/`form.ts` now or just the folder structure
- Test organization details — how to migrate co-located tests, test runner config changes
- CLI command surface — whether to keep `hr-auto extract` or restructure to `hr-auto onboarding extract`
- Import path conventions and barrel exports

</decisions>

<specifics>
## Specific Ideas

- Key design principle: "Adding offboarding later would just mean a new `workflows/offboarding/` folder with its own schema and field mappings, reusing `crm/search`, `crm/extract`, `ucpath/navigate`, etc."
- Shared modules (auth, browser, crm, ucpath, utils) must be workflow-agnostic
- Each workflow composes shared pieces with its own schema, UCPath form target, and orchestration logic

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `extractField(page, label)` in `src/onboarding/extract.ts` — 3-strategy field reader, moves to `crm/extract.ts`
- `searchByEmail`, `selectLatestResult` in `src/onboarding/search.ts` — already generic, move to `crm/`
- `navigateToEntrySheet` in `src/onboarding/navigate.ts` — needs generalization to `navigateToSection(page, sectionName)`
- `EmployeeDataSchema` in `src/onboarding/schema.ts` — moves to `workflows/onboarding/schema.ts`

### Established Patterns
- PII-safe logging (never log extracted values)
- `FIELD_MAP` pattern with label variants for flexible Salesforce DOM extraction
- Separate browser contexts per system (UCPath vs ACT CRM)
- `validateEnv()` early fail pattern

### Integration Points
- `src/cli.ts` imports from `src/onboarding/index.ts` — needs rewiring to `src/workflows/onboarding/index.ts`
- Co-located tests (`schema.test.ts`, `env.test.ts`, `log.test.ts`) need migration to `tests/`
- `src/workflow/index.ts` exists but is empty — delete after restructure

</code_context>
