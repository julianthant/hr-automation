# Phase 01.1: Modular Codebase Restructure - Research

**Researched:** 2026-03-14
**Domain:** TypeScript project restructuring, module organization, test migration
**Confidence:** HIGH

## Summary

This phase is a pure structural refactor of an existing, working TypeScript codebase. No new functionality is added. The goal is to extract shared CRM and UCPath modules from the flat `src/onboarding/` directory into reusable `src/crm/` and `src/ucpath/` modules, move onboarding-specific code into `src/workflows/onboarding/`, and relocate co-located tests from `src/` into a top-level `tests/` hierarchy.

The codebase is small (18 TypeScript source files, 13 passing tests) and uses `tsx` for both running and testing (Node's built-in `node:test` runner). Since `tsx` compiles on-the-fly without writing to disk, the restructure is primarily about (a) moving files, (b) updating import paths, and (c) adjusting the `tsconfig.json` and `package.json` test script to accommodate the new layout.

**Primary recommendation:** Execute the restructure in a strict sequence -- create new directories first, move shared modules (crm), then move workflow-specific code (workflows/onboarding), update all imports, adjust config files, run tests after each major move to catch breakage immediately.

### Pattern 1: Extract Shared Module (CRM)

**What:** Take functions that are generic CRM operations and move them to `src/crm/`, leaving onboarding-specific knowledge in `src/workflows/onboarding/`.

**When to use:** Any function that operates on CRM page structure without knowing which workflow is running.

**Boundary rule:** If a function needs to know field names specific to onboarding (like FIELD_MAP), it belongs in `workflows/onboarding/`. If it operates on any label string passed to it, it belongs in `crm/`.

**Example -- crm/extract.ts (generic):**
```typescript
import type { Page, Locator } from "playwright";
import { log } from "../utils/log.js";

/**
 * Try to extract a single field value from the page using multiple strategies.
 * Generic CRM extraction -- does not know which fields to extract.
 */
export async function extractField(
  page: Page,
  label: string,
): Promise<string | null> {
  // Strategy 1: label-based
  const byLabel: Locator = page
    .locator(`text="${label}"`)
    .locator("xpath=..")
    .locator("span, dd, td")
    .last();

  // Strategy 2: ARIA
  const byAria: Locator = page.getByLabel(label);

  // Strategy 3: table cell
  const byTableCell: Locator = page
    .locator(`td:has-text("${label}")`)
    .locator("xpath=following-sibling::td[1]");

  for (const locator of [byLabel, byAria, byTableCell]) {
    try {
      const text = await locator.first().textContent({ timeout: 3_000 });
      if (text && text.trim()) {
        return text.trim();
      }
    } catch {
      continue;
    }
  }
  return null;
}
```

**Example -- workflows/onboarding/extract.ts (onboarding-specific):**
```typescript
import type { Page } from "playwright";
import { log } from "../../utils/log.js";
import { extractField } from "../../crm/extract.js";

const FIELD_MAP: Record<string, string[]> = {
  positionNumber: ["Position Number", "Position #", "Position No"],
  firstName: ["First Name", "Legal First Name"],
  // ... all onboarding-specific field mappings
};

export async function extractRawFields(
  page: Page,
): Promise<Record<string, string | null>> {
  const raw: Record<string, string | null> = {};
  for (const [field, labels] of Object.entries(FIELD_MAP)) {
    log.step(`Extracting ${field}...`);
    let value: string | null = null;
    for (const label of labels) {
      value = await extractField(page, label);
      if (value) break;
    }
    raw[field] = value;
  }
  return raw;
}
```

### Pattern 2: Generalize Navigation

**What:** Replace hardcoded `navigateToEntrySheet` with parameterized `navigateToSection(page, sectionName)`.

**Example -- crm/navigate.ts:**
```typescript
import type { Page } from "playwright";
import { log } from "../utils/log.js";

/**
 * Navigate from an employee record page to a named section.
 * Generic -- each workflow passes its own section name.
 */
export async function navigateToSection(
  page: Page,
  sectionName: string,
): Promise<void> {
  log.step(`Looking for ${sectionName}...`);

  const link = page
    .getByRole("link", { name: new RegExp(sectionName, "i") })
    .or(page.getByText(sectionName))
    .or(page.getByRole("tab", { name: new RegExp(sectionName, "i") }));

  await link.first().click({ timeout: 10_000 });

  log.step(`Waiting for ${sectionName} to load...`);
  await page.waitForLoadState("networkidle", { timeout: 15_000 });

  log.step(`${sectionName} loaded`);
}
```

The onboarding workflow calls it as: `navigateToSection(page, "UCPath Entry Sheet")`.

### Pattern 3: Barrel Exports for Small Module APIs

**What:** Use `index.ts` barrel files at module boundaries (crm, workflows/onboarding) to define the public API of each module.

**When to use:** This project is small (~20 files). Barrel files are appropriate here because: (a) each module has a small, stable API (3-5 exports), (b) there are no bundling/tree-shaking concerns (CLI tool, not a library), (c) they clearly communicate what is "public" from each module.

**When NOT to use:** Do not create deep nesting or barrel chains. No barrel for `utils/` or `auth/` (they already work with direct imports and are small).

**Example -- crm/index.ts:**
```typescript
export { searchByEmail, selectLatestResult } from "./search.js";
export { navigateToSection } from "./navigate.js";
export { extractField } from "./extract.js";
export { ExtractionError } from "./types.js";
```

**Example -- workflows/onboarding/index.ts:**
```typescript
export { extractRawFields } from "./extract.js";

export type { EmployeeData } from "./schema.js";
```

### Pattern 4: Test Organization with Separate tests/ Directory

**What:** Move co-located `.test.ts` files from `src/` to `tests/unit/` and `tests/e2e/`.

**Key technical detail:** The current `tsconfig.json` has `rootDir: "src"` and `include: ["src/**/*"]`. Tests in `tests/` are outside this scope. Two approaches:

**Recommended approach (simpler for this project):**
- Keep `tsconfig.json` for source compilation (`rootDir: "src"`)
- Create `tsconfig.test.json` that extends `tsconfig.json` but with broader include
- Update `package.json` test script to use the test tsconfig (not needed for tsx --test, which doesn't use tsc)
- The `tsx --test` command does NOT require tsconfig alignment -- it compiles on-the-fly

**tsconfig.test.json:**
```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "rootDir": "."
  },
  "include": ["src/**/*", "tests/**/*"]
}
```

**Updated package.json test script:**
```json
{
  "scripts": {
    "test": "tsx --test tests/**/*.test.ts",
    "typecheck": "tsc --noEmit",
    "typecheck:all": "tsc --noEmit -p tsconfig.test.json"
  }
}
```

### Anti-Patterns to Avoid

- **Circular imports between crm/ and workflows/:** The dependency must be one-way: `workflows/onboarding/ -> crm/`, never `crm/ -> workflows/onboarding/`. CRM modules must not know about specific workflows.
- **Moving ExtractionError to utils/:** It is a CRM-level concern (extraction can fail). It belongs in `crm/types.ts`, not in a generic utils bucket.
- **Putting FIELD_MAP in crm/:** FIELD_MAP contains onboarding-specific field knowledge. It belongs in `workflows/onboarding/extract.ts`.
- **Giant commit with all moves at once:** Each major move should be verifiable (tests pass) before the next. If something breaks, the blast radius is one step.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Updating import paths | Regex-based find/replace across files | Manual review of each import in each moved file | Only ~10 files have imports that change; manual is safer and faster than a script that might miss `.js` extensions |
| Test discovery | Custom test glob scripts | `tsx --test tests/**/*.test.ts` | Built-in Node test runner with tsx handles this |
| Type checking tests | Running tsc on test files with production tsconfig | `tsconfig.test.json` extending base config | Keeps production build clean while enabling type checking of tests |

**Key insight:** This is a small codebase (18 files, 13 tests). The "right" amount of tooling for a restructure this size is zero new tools -- just move files, fix imports, verify tests pass.

## Common Pitfalls

### Pitfall 1: Forgetting .js Extensions in ESM Imports
**What goes wrong:** TypeScript ESM modules require `.js` extensions in import paths even though the source files are `.ts`. After moving files, you write `import { x } from "../crm/extract"` instead of `import { x } from "../crm/extract.js"`.
**Why it happens:** Muscle memory from CommonJS or bundler-based projects where extensions are optional.
**How to avoid:** Every import statement in this project already uses `.js` extensions. After moving files, ensure all updated imports retain the `.js` extension.
**Warning signs:** `ERR_MODULE_NOT_FOUND` at runtime, or TypeScript errors about not finding modules.

### Pitfall 2: Breaking the Existing Test Imports
**What goes wrong:** Tests currently import from relative paths like `./schema.js` and `./types.js`. After moving to `tests/unit/`, these paths are completely wrong.
**Why it happens:** Tests are written with co-located imports, not cross-directory imports.
**How to avoid:** When moving `schema.test.ts` to `tests/unit/`, update its imports from `./schema.js` to `../../src/workflows/onboarding/schema.js` (or use the barrel: `../../src/workflows/onboarding/index.js`). Similarly for `env.test.ts` and `log.test.ts`.
**Warning signs:** Tests fail to compile or import errors at runtime.

### Pitfall 3: Deleting Old Directories Before Verifying
**What goes wrong:** You delete `src/onboarding/` before confirming all code has been moved and all imports updated. Something still references the old path.
**Why it happens:** Eagerness to clean up.
**How to avoid:** Move files, update imports, run `tsc --noEmit` AND `tsx --test`, only THEN delete old directories. Typecheck catches stale imports.
**Warning signs:** TypeScript errors about missing modules after deletion.

### Pitfall 4: CLI Import Path Not Updated
**What goes wrong:** `src/cli.ts` currently imports from `./onboarding/index.js`. After restructure, this path no longer exists but the error only surfaces at runtime, not compile time (if the old directory still exists during transition).
**Why it happens:** CLI is the entry point and easy to forget in a batch of module moves.
**How to avoid:** Update `src/cli.ts` imports early -- it's the only consumer of the onboarding barrel. Change to `./workflows/onboarding/index.js` and `./crm/index.js`.
**Warning signs:** CLI commands fail at runtime with "Cannot find module" errors.

### Pitfall 5: ExtractionError Import Chain
**What goes wrong:** `ExtractionError` is currently in `src/onboarding/types.ts` and imported by both `search.ts` (moving to crm/) and `schema.ts` (moving to workflows/onboarding/). After the split, both modules need to import from `crm/types.ts`, but if you move search.ts before moving ExtractionError, you break the intermediate state.
**Why it happens:** ExtractionError has dependents in both the CRM and workflow layers.
**How to avoid:** Move `ExtractionError` to `crm/types.ts` FIRST, then move the files that import it. This way the dependency is stable during the transition.
**Warning signs:** Multiple files fail to compile simultaneously.

## Discretion Recommendations

### UCPath Module Scope
**Recommendation:** Keep `src/ucpath/index.ts` as a stub file (it already exists with a placeholder comment). Do NOT create `navigate.ts` or `form.ts` stubs yet -- they would contain zero real code and Phase 3 (UCPath Entry) will define them based on actual PeopleSoft DOM discovery. Creating empty stubs now adds files that might be structured wrong once real requirements land.

### Test Organization
**Recommendation:** Use the layout from the locked decisions: `tests/unit/` for pure logic tests (schema, env, log) and `tests/e2e/` as an empty directory placeholder. Update the package.json test script glob to `tests/**/*.test.ts`. Create `tsconfig.test.json` so `tsc --noEmit -p tsconfig.test.json` can type-check tests too.

### CLI Command Surface
**Recommendation:** Keep `hr-auto extract` (not `hr-auto onboarding extract`) for now. The user is the only consumer, there is only one workflow, and the shorter command is more ergonomic. When a second workflow is added (v2), that is the right time to introduce subcommands like `hr-auto onboarding extract` vs `hr-auto offboarding run`. Premature namespacing adds friction with no benefit.

### Import Path Conventions
**Recommendation:** Use direct file imports with `.js` extensions (matching the existing convention). Use barrel `index.ts` files only at module boundaries (`crm/index.ts`, `workflows/onboarding/index.ts`) to define the public API. External consumers (cli.ts, tests) import from the barrel. Internal module files import directly from siblings. This matches the existing `src/onboarding/index.ts` pattern.
