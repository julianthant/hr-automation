# Phase 3.1: CRM Additional Fields and Onboarding Tracker - Research

**Researched:** 2026-03-15
**Domain:** CRM field extraction (Salesforce Visualforce), Excel spreadsheet generation (Node.js)
**Confidence:** HIGH

## Summary

This phase has two distinct deliverables: (1) extract two additional fields from the ACT CRM record page -- department number and recruitment number -- before navigating to the UCPath Entry Sheet, and (2) create an onboarding tracking spreadsheet that aggregates all employee data including rehire status and workflow progress.

The CRM extraction work follows the same established pattern used in Phase 2: the `extractField` function in `src/crm/extract.ts` already handles Visualforce label-to-value extraction. The new fields live on the CRM record page (the page reached after `selectLatestResult`, before `navigateToSection("UCPath Entry Sheet")`). Department number requires a regex parse from parenthesized text (e.g., "(000412)" from the department field value). Recruitment number is a separate field on the same page. Both fields need live selector discovery since the exact label text is unknown until verified against the live CRM.

For the spreadsheet, ExcelJS is the standard Node.js library for creating and updating .xlsx files. It has built-in TypeScript types, supports read-modify-write workflows (essential for reusing the spreadsheet across multiple employees in a session), and is the most actively maintained option in the ecosystem.

**Primary recommendation:** Add department number and recruitment number to the EmployeeData schema (optional fields), extract them from the CRM record page using the existing `extractField` pattern, create a `src/tracker/` module using ExcelJS for spreadsheet management, and integrate the tracker into the `start-onboarding` CLI workflow.

### Pattern 1: CRM Record Page Extraction (Before UCPath Entry Sheet)
**What:** Extract department number and recruitment number from the CRM record page, which is the intermediate page between search results and the UCPath Entry Sheet.
**When to use:** After `selectLatestResult()` returns and before `navigateToSection("UCPath Entry Sheet")` is called.
**Example:**
```typescript
// In the start-onboarding workflow, the flow is:
// 1. searchByEmail(page, email)
// 2. selectLatestResult(page)
// 3. >>> NEW: extractRecordPageFields(page) <<<  -- extracts dept#, recruitment#
// 4. navigateToSection(page, "UCPath Entry Sheet")
// 5. extractRawFields(page)  -- extracts employee data fields

// The record page URL pattern after selectLatestResult is:
// https://act-crm.my.site.com/hr/ONB_ViewOnboarding?id=RECORD_ID
// Department and recruitment number are on THIS page, not the Entry Sheet.
```

### Pattern 2: Department Number Regex Extraction
**What:** The department field on the CRM record page contains a department name with a number in parentheses, e.g., "Computer Science (000412)". The department number must be extracted via regex.
**When to use:** After extracting the raw department text value.
**Example:**
```typescript
// Source: Phase 3.1 success criteria and project_onboarding_workflow.md memory
function parseDepartmentNumber(deptText: string): string | null {
  const match = deptText.match(/\((\d+)\)/);
  return match ? match[1] : null;
}

// Examples:
// "Computer Science (000412)" -> "000412"
// "Biology (000301)" -> "000301"
// "Unknown Department" -> null (no parens)
```

### Pattern 3: ExcelJS Read-Modify-Write for Session Reuse
**What:** The spreadsheet must persist across multiple employees processed in the same session. ExcelJS supports reading an existing file, adding rows, and writing back.
**When to use:** Every time an employee is processed -- first employee creates the file, subsequent employees append.
**Example:**
```typescript
// Source: ExcelJS GitHub README
import ExcelJS from "exceljs";

async function appendToTracker(filePath: string, rowData: Record<string, string>): Promise<void> {
  const workbook = new ExcelJS.Workbook();

  try {
    // Try to read existing file
    await workbook.xlsx.readFile(filePath);
  } catch {
    // File does not exist -- will create fresh
  }

  let sheet = workbook.getWorksheet("Onboarding Tracker");
  if (!sheet) {
    sheet = workbook.addWorksheet("Onboarding Tracker");
    // Set up columns on first creation
    sheet.columns = [
      { header: "First Name", key: "firstName", width: 15 },
      { header: "Last Name", key: "lastName", width: 15 },
      // ... more columns
    ];
  }

  sheet.addRow(rowData);
  await workbook.xlsx.writeFile(filePath);
}
```

### Pattern 4: SSN Masking for Spreadsheet
**What:** SSN must be stored in masked format in the spreadsheet (PII safety). The success criteria says "SSN (masked)".
**When to use:** When writing employee data to the tracker.
**Example:**
```typescript
// Mask SSN: "123-45-6789" -> "XXX-XX-6789"
function maskSsn(ssn: string | undefined): string {
  if (!ssn) return "N/A";
  return ssn.replace(/^\d{3}-\d{2}/, "XXX-XX");
}
```

### Anti-Patterns to Avoid
- **Extracting new fields from the Entry Sheet page:** Department number and recruitment number are on the CRM record page (ViewOnboarding), NOT the UCPath Entry Sheet (ONB_PPSEntrySheet). The workflow must extract them BEFORE navigating to the Entry Sheet.
- **Storing raw SSN in the spreadsheet:** SSN must always be masked. The project has strict PII safety rules -- never log or store raw SSN.
- **Creating a new spreadsheet file per employee:** The tracker must be a single file that accumulates rows across the session. Use read-modify-write, not create-per-employee.
- **Putting tracker logic in the workflow module:** The tracker should be a shared module (`src/tracker/`) since Phase 3.2 (I9) and Phase 5 (batch) will also need to update it.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Excel file generation | Custom XML/ZIP manipulation | ExcelJS | .xlsx is a complex zipped XML format; hand-rolling is error-prone and fragile |
| Date formatting in Excel | String dates in cells | ExcelJS date formatting | ExcelJS handles Excel serial dates properly; string dates cause sorting/filtering issues |
| Column auto-width | Manual width calculation | ExcelJS column width property | Set reasonable fixed widths; auto-width requires measuring text which adds complexity for no benefit |

**Key insight:** The spreadsheet is a simple append-only tracker with fixed columns. ExcelJS is more than capable but resist the urge to over-engineer formatting, conditional styles, or formulas. The value is in the data, not the presentation.

## Common Pitfalls

### Pitfall 1: Extracting from the Wrong CRM Page
**What goes wrong:** Trying to extract department number or recruitment number from the UCPath Entry Sheet (ONB_PPSEntrySheet) instead of the CRM record page (ONB_ViewOnboarding).
**Why it happens:** The current workflow goes straight from `selectLatestResult` to `navigateToSection("UCPath Entry Sheet")`. Developers might assume the new fields are on the Entry Sheet.
**How to avoid:** Extract record page fields AFTER `selectLatestResult` and BEFORE `navigateToSection`. The record page URL contains `ViewOnboarding` in the path.
**Warning signs:** Fields not found, or `extractField` returning null for department/recruitment labels.

### Pitfall 2: ExcelJS File Locking on Windows
**What goes wrong:** `writeFile` fails because the .xlsx file is open in Excel.
**Why it happens:** Windows file locking prevents writing to a file that is open in another application.
**How to avoid:** Log a clear error message when write fails. Do not attempt to forcibly close the file. The user should close the spreadsheet before running the tool.
**Warning signs:** EBUSY or EPERM errors on writeFile.

### Pitfall 3: Department Number Regex Edge Cases
**What goes wrong:** The regex `\((\d+)\)` might match the wrong parenthesized number if the department text contains multiple sets of parentheses.
**Why it happens:** Some department names might have explanatory text in parentheses before the actual number.
**How to avoid:** Use a more specific regex that targets the department number format (typically 6 digits): `\((\d{4,6})\)`. Also, extract the LAST match if multiple are found, since the department number is typically at the end.
**Warning signs:** Extracted number does not look like a department number (too short, too long, or contains non-numeric characters).

### Pitfall 4: Schema Change Breaking Existing Tests
**What goes wrong:** Adding new optional fields to EmployeeDataSchema causes existing schema tests to need updates.
**Why it happens:** The VALID_DATA fixture in schema.test.ts may need updating, or tests checking field counts break.
**How to avoid:** New fields should be optional (`z.string().optional()`), which means existing tests with VALID_DATA that omit them should still pass. Verify tests pass after schema changes.
**Warning signs:** Schema test failures after adding fields.

### Pitfall 5: Spreadsheet File Path Assumptions
**What goes wrong:** Using a hardcoded file path that doesn't work across environments or when run from different working directories.
**Why it happens:** The tool is run via npm scripts which set cwd to project root, but the tracker file should go in a user-accessible location.
**How to avoid:** Use a configurable output path with a sensible default (e.g., `./onboarding-tracker.xlsx` in the current working directory). Consider making it a CLI option in Phase 5.
**Warning signs:** File not found errors or files created in unexpected locations.

### Extracting Record Page Fields (New Function)
```typescript
// New function in src/crm/extract.ts or src/workflows/onboarding/extract.ts
// Extracts fields from the CRM record page (ViewOnboarding)
// BEFORE navigating to UCPath Entry Sheet
export async function extractRecordPageFields(
  page: Page,
): Promise<{ departmentNumber: string | null; recruitmentNumber: string | null }> {
  // Extract raw department text (includes number in parens)
  const deptText = await extractField(page, "Department");
  const recruitmentNumber = await extractField(page, "Recruitment Number");

  // Parse department number from parentheses
  let departmentNumber: string | null = null;
  if (deptText) {
    const match = deptText.match(/\((\d{4,6})\)/);
    departmentNumber = match ? match[1] : null;
  }

  return { departmentNumber, recruitmentNumber };
}
```

### ExcelJS Tracker Module
```typescript
// Source: ExcelJS GitHub README (https://github.com/exceljs/exceljs)
import ExcelJS from "exceljs";

const TRACKER_COLUMNS = [
  { header: "First Name", key: "firstName", width: 15 },
  { header: "Last Name", key: "lastName", width: 15 },
  { header: "SSN (Masked)", key: "ssnMasked", width: 15 },
  { header: "DOB", key: "dob", width: 12 },
  { header: "Dept #", key: "departmentNumber", width: 10 },
  { header: "Recruitment #", key: "recruitmentNumber", width: 15 },
  { header: "Rehire", key: "rehire", width: 8 },
  { header: "Effective Date", key: "effectiveDate", width: 14 },
  { header: "CRM Extracted", key: "crmExtracted", width: 14 },
  { header: "Person Search", key: "personSearch", width: 14 },
  { header: "Transaction", key: "transaction", width: 14 },
];

export async function updateTracker(
  filePath: string,
  data: TrackerRow,
): Promise<void> {
  const workbook = new ExcelJS.Workbook();

  try {
    await workbook.xlsx.readFile(filePath);
  } catch {
    // File does not exist yet -- fresh workbook
  }

  let sheet = workbook.getWorksheet("Onboarding Tracker");
  if (!sheet) {
    sheet = workbook.addWorksheet("Onboarding Tracker");
    sheet.columns = TRACKER_COLUMNS;
    // Bold header row
    sheet.getRow(1).font = { bold: true };
  }

  sheet.addRow(data);
  await workbook.xlsx.writeFile(filePath);
}
```

## Open Questions

1. **CRM Record Page Field Labels**
   - What we know: Department text field contains the department name with number in parentheses. Recruitment number is a separate field. Both are on the ViewOnboarding page.
   - What's unclear: The exact label text for these fields on the CRM page (e.g., "Department" vs "Department Name" vs "Dept"). The extraction strategy with `extractField` tries multiple label variants, but the actual labels need live verification.
   - Recommendation: Use best-guess labels in the FIELD_MAP and verify via live selector discovery (the project's established debug loop pattern).

2. **Tracker File Location**
   - What we know: The spreadsheet needs to persist across the session and be accessible to the user.
   - What's unclear: Whether the user wants it in the project root, a specific output directory, or configurable via CLI flag.
   - Recommendation: Default to `./onboarding-tracker.xlsx` in the current working directory. Can be made configurable in Phase 5 (batch CLI).

3. **Workflow Status Column Values**
   - What we know: Success criteria says "workflow step status" -- the tracker needs columns for each workflow step's status.
   - What's unclear: The exact set of status values (e.g., "Done", "Skipped", "Failed", "Pending").
   - Recommendation: Use simple status strings: "Done", "Failed", "Skipped", "Pending". Keep it human-readable since this is an HR tool.
