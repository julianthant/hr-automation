  <action>
    This task requires a live authenticated ACT CRM session. The user must have run `npm run test-login` successfully before this task begins.

    **Step 1: Ensure auth session exists**
    - Run `npm run test-login` if needed (user handles Duo MFA approval)
    - Verify .auth/actcrm-state.json exists

    **Step 2: Run extraction against a test employee email**
    - Run `npm run extract -- <test-email>` (user will provide the email)
    - Observe where it fails -- this is the selector discovery process

    **Step 3: Fix selectors iteratively**
    Following the Phase 1 pattern (where 6/7 deviations were selector fixes found during live testing):

    For search.ts (EXTR-01):
    - If search input selector fails: inspect the ACT CRM page to find the actual search element (may be a Lightning input, a global search bar, or a list view filter). Update the selector chain.
    - If search submission fails: check if Enter key works or if a search button needs clicking. Update accordingly.
    - If results do not load: check wait condition -- may need waitForSelector on a specific result container instead of networkidle.

    For search.ts selectLatestResult (EXTR-02):
    - If row selector fails: inspect the actual results layout (table, list, cards). Update row locator.
    - If date extraction fails: find the actual date column/element position. Update date cell selector.
    - If date parsing fails: observe the actual date format (could be "Mar 13, 2026" or "2026-03-13" or "03/13/2026"). Adjust Date parsing.

    For navigate.ts (EXTR-03):
    - If "UCPath Entry Sheet" link not found: inspect the profile page to find the actual link/tab/button text and element type. Update selector.
    - If navigation does not complete: check if it opens in an iframe, a new tab, or an in-page section. Adjust wait strategy.

    For extract.ts (EXTR-04):
    - For each of the 10 fields: if extractField returns null, inspect the entry sheet page layout to determine:
      a. What HTML structure holds the field (dt/dd pairs, table rows, Lightning record sections, custom components)
      b. What the exact label text is (may differ from FIELD_MAP guesses)
      c. Whether the value is in a span, input, td, or other element
    - Update FIELD_MAP with correct label text discovered from the live page
    - Update extractField strategies to match the actual DOM structure
    - If SSN is partially masked on the page: this is a blocking issue. Log it and stop -- cannot extract masked data.

    For schema.ts (EXTR-05):
    - If date format from the live page does not match the schema regex: update effectiveDate validation to accept the actual format observed
    - If SSN format differs from XXX-XX-XXXX: update ssn validation to match reality
    - After schema changes, re-run unit tests to ensure they still pass (update test data if format expectations changed)

    **Step 4: Re-run full pipeline**
    - Run `npm run extract -- <test-email>` again after fixes
    - Repeat fix cycle until the full pipeline succeeds: search -> select latest -> navigate -> extract all 10 fields -> Zod validation passes

    **Step 5: Verify all tests still pass**
    - Run `npx tsx --test src/**/*.test.ts && npx tsc --noEmit`
    - If schema.ts was modified, ensure schema.test.ts test data matches the updated validation rules

    IMPORTANT: Every selector change must retain the `// SELECTOR: adjusted after live testing` comment pattern (replacing "may need adjustment").
    IMPORTANT: Do NOT log any extracted PII values during debugging. Use page.screenshot() if visual inspection is needed.
