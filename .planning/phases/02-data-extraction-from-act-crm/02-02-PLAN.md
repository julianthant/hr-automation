  <action>
Run the extraction pipeline against a live ACT CRM session and iteratively fix selectors until all 10 fields extract successfully. This is the core work of this plan.

**Step-by-step process:**

1. **Authenticate first:** Run `npm run test-login` to establish a live ACT CRM session. Wait for Julian to approve the Duo MFA push.

2. **Run initial extraction attempt:** Run `npm run extract -- <test-email>` (Julian will provide the test email, or use one from a recent onboarding). Observe what happens -- which step fails first.

3. **Fix selectors iteratively, one module at a time:**

   **Search (src/crm/search.ts):**
   - If searchByEmail fails: Take a screenshot (`page.screenshot()`), inspect the page DOM via `page.content()` or `page.locator('*').evaluateAll(...)` to find the actual search input element. Update the selector chain in `searchByEmail`.
   - If selectLatestResult fails: Inspect the search results DOM structure. The results may not be a `<table>` -- could be Lightning list items, cards, or a custom component. Update the row locator and date cell selector. Verify which column contains dates.
   - After each fix, re-run `npm run extract -- <email>` to confirm the fix works before moving to the next module.

   **Navigate (src/crm/navigate.ts):**
   - If navigateToSection("UCPath Entry Sheet") fails: Inspect the employee record page DOM. The section may be a Lightning tab, a related list link, a button, or a sidebar item. Update the locator chain.
   - Check if the section name is exactly "UCPath Entry Sheet" or something slightly different (e.g., "UCPath Entry", "Entry Sheet", etc.).
   - Re-run to confirm navigation lands on the correct page.

   **Extract (src/crm/extract.ts + src/workflows/onboarding/extract.ts):**
   - If extractField returns null for fields: Inspect the UCPath Entry Sheet DOM to determine the actual field layout. It could be:
     - Salesforce record detail layout (label/value pairs in `lightning-output-field`)
     - Custom Visualforce table
     - Plain HTML table with th/td or dt/dd
   - Update the extraction strategies in `extractField` to match the actual DOM pattern found.
   - Update the FIELD_MAP label variants in `src/workflows/onboarding/extract.ts` to match the exact label text on the page.
   - Test each field individually if needed. For PII fields (SSN), verify the display format (full vs masked) -- if masked, this is a blocking issue to surface to Julian.

4. **Debugging techniques (use as needed):**
   - `await page.screenshot({ path: 'debug-search.png' })` -- capture visual state
   - `const html = await page.content()` -- get full page HTML for inspection
   - `await page.locator('body').evaluate(el => el.innerHTML.substring(0, 5000))` -- targeted DOM inspection
   - `console.log(await page.url())` -- verify current page URL
   - Add temporary `await page.pause()` to freeze the browser for manual DevTools inspection (remove before committing)

5. **Key constraints:**
   - Do NOT log extracted PII values. When debugging, log field names and whether extraction succeeded/failed, not the actual values.
   - All selector fixes must keep the `// SELECTOR:` comment markers (updated to note "adjusted from live testing").
   - Preserve the 3-strategy fallback pattern in extractField unless the actual DOM makes it unnecessary.
   - Use CSS/text-based Playwright selectors only -- never XPath for Salesforce shadow DOM (it does not pierce shadow boundaries).

6. **After all selectors work:** Run the full pipeline once more to confirm end-to-end success: search -> navigate -> extract -> validate passes with all 10 fields.

<task type="auto">
  <name>Task 2: Tighten Zod schema to match observed data formats</name>
  <files>src/workflows/onboarding/schema.ts, tests/unit/schema.test.ts</files>
  <action>
After Task 1 succeeds and real data has been observed, tighten the Zod schema validators that were left loose (per Plan 02-01 decisions):

1. **effectiveDate:** Currently `min(1)` (deliberately loose per decision). Now that the actual ACT CRM date format is known, add a regex or format constraint matching the observed format. For example, if dates appear as "MM/DD/YYYY", use `.regex(/^\d{2}\/\d{2}\/\d{4}$/)`. If multiple formats are seen, use a regex union.

2. **state:** Currently `min(1)` (deliberately loose per decision). If all observed states are 2-letter codes (e.g., "CA"), tighten to `.length(2)`. If full state names appear (e.g., "California"), keep `min(1)`.

3. **Other fields:** If any other field shows a consistent format in the live data (e.g., positionNumber is always 8 digits, wage always has a dollar sign or decimal), add appropriate format constraints. Only tighten what the live data justifies -- do not add speculative constraints.

4. **Update unit tests in tests/unit/schema.test.ts:**
   - Update the valid test data to match the actual formats observed (do NOT use real PII -- use realistic fake data in the same format).
   - Add test cases for the new format constraints (e.g., if effectiveDate now requires MM/DD/YYYY, test that "2026-03-14" is rejected and "03/14/2026" passes).
   - Ensure existing tests still pass or are updated to match the tightened schema.

5. **Run the full test suite** to confirm no regressions.

Do NOT change any field names or the overall schema shape -- only tighten validators based on observed data.
