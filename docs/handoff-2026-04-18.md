#### Task 1 — Aggregate stats panel (improvements §4.5)

**Backend** (`src/tracker/dashboard.ts` + sibling helper):
- `GET /api/stats?days=7` returns aggregated stats per workflow over the last N days.

  Response shape:
  ```ts
  {
    windowDays: number,
    windowStart: string,  // ISO
    windowEnd: string,    // ISO
    byWorkflow: Record<string, {
      total: number,
      succeeded: number,
      failed: number,
      skipped: number,
      successRate: number,        // 0..1
      averageDurationMs: number,  // wall-clock first-running → done/failed
      slowestStep: { step: string, averageMs: number } | null,
      lastRun: string             // ISO of most recent run end
    }>
  }
  ```
- Use the `buildStatsHandler({ readDates, readEntries })` factory pattern (see `buildSearchHandler` in `src/tracker/dashboard.ts` for shape).
- Walk JSONL across each date in the window per workflow. Per (id, runId): sort by ts; first running ts = start, last done/failed ts = end. Average duration over completed runs only. Slowest step = max average ms across step transitions. There's already a `computeStepDurations` helper in `src/tracker/dashboard.ts` you can reuse.
- Test: `tests/unit/tracker/stats-endpoint.test.ts` — fake JSONL across 3 days, assert math (≥4 tests).

**Frontend** (`src/dashboard/components/StatsPanel.tsx`):
- Collapsible (default expanded). Header has a `[7d / 14d / 30d]` toggle.
- Per workflow row: name + total + success rate (% with thin bar) + average duration + slowest step + last-run timestamp (relative).
- **Match existing dashboard aesthetic** — inspect `StatPills.tsx`, `QueuePanel.tsx`, `EntryItem.tsx` for typography/colors/spacing. Reuse HeroUI `Card` / `Chip` / `Progress`. Do NOT use the runner drawer's amber/Instrument Serif/Geist Mono — those are reserved for the runner.
- Empty state: "No runs in the last <N> days" centered, muted.
- Mount it sensibly — probably above the queue panel or as a tab. Inspect `src/dashboard/main.tsx` for layout context.

**Commit:** `feat(dashboard): aggregate stats panel — runs, success rate, slowest step`

#### Task 2 — Run diff view (improvements §4.7)

**Backend:**
- `GET /api/diff?workflow=X&id=Y&runA=<runId>&runB=<runId>` returns:
  ```ts
  {
    workflow: string,
    id: string,
    runA: { runId, status, step?, data, ts },
    runB: { runId, status, step?, data, ts },
    diff: Array<{ key: string, a: string|null, b: string|null, changed: boolean }>
  }
  ```
- Fetch each run's most-recent tracker entry by walking `readEntriesForDate` backward from today until both runIds are found (or 30 days exhausted → 404).
- Diff: union of `data` keys from both runs; for each key emit `{ a, b, changed: a !== b }`. Include unchanged rows for context.
- `buildDiffHandler({ readDates, readEntries })` factory; tests in `tests/unit/tracker/diff-endpoint.test.ts` cover added/removed/changed/same/missing-run-404 cases.

**Frontend** (`src/dashboard/components/RunDiffView.tsx`):
- Inline expansion under an entry row (NOT a modal — keeps dashboard flow). Triggered by a "Compare runs" link in `RunSelector.tsx` (or wherever runs are listed). Only visible when the entry has ≥2 runs. Default: latest two runs.
- Two-column layout: runA on left, runB on right. Each row: `key | a-value | b-value`. Changed rows highlighted using HeroUI's semantic color (`success`/`warning`) — NOT custom colors.

**Commit:** `feat(dashboard): run diff view — side-by-side comparison of two runs of the same id`

## Reference patterns (read before writing similar code)

### Backend endpoint factories
Look at `src/tracker/dashboard.ts` for patterns like `buildScreenshotsHandler`, `buildSelectorWarningsHandler`, `buildSearchHandler`, `buildSchemaHandler`, `buildRunHandler`, `buildCancelHandler`, `buildActiveRunsHandler`. They all take a `deps` object so unit tests can inject mocks. Pattern your new handlers the same way.

### Dashboard frontend conventions
- HeroUI v3 + Tailwind. NO new motion library — Tailwind transitions + HeroUI built-ins only.
- Components in `src/dashboard/components/`. Hooks in `src/dashboard/components/hooks/` and `src/dashboard/workflows-context.tsx`.
- Type-aware tracker value rendering: `src/dashboard/components/types.ts` has `formatTrackerValue` — reuse it.
- Existing dashboard aesthetic is dark-themed, unobtrusive, mono+sans pair already in use. Inspect first.
- The runner drawer (`RunnerDrawer.tsx`) intentionally uses a separate aesthetic (Instrument Serif + Geist Mono + amber). Don't copy that to other panels.

### Test conventions
- `node:test` + `node:assert/strict`. Mirror the source layout: `src/foo/bar.ts` → `tests/unit/foo/bar.test.ts`.
- Frontend React components are NOT unit-tested per `tests/CLAUDE.md`. Verification is `npm run build:dashboard` + manual smoke.
- Backend handlers ARE unit-tested via the factory + injected deps pattern.

### Commit conventions
- HEREDOC with `Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>`.
- One logical change per commit.
- Conventional Commits prefixes: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`.
- NO `--no-verify`, NO `git add -A`, NO `--amend`.
