### Selector intelligence loop (per-system CLAUDE.md tells you this)

Before mapping any new selector:
1. `npm run selector:search "<intent>"` — does a match exist? If yes, USE IT.
2. Check the per-system `LESSONS.md` for related failure modes.
3. Map a new selector via `playwright-cli`, add JSDoc + `@tags` + `// verified <date>` in `selectors.ts`.
4. Run `npm run selectors:catalog` to regenerate `SELECTORS.md`.
5. If you discovered a non-obvious failure mode, append a lesson to `LESSONS.md` (template in any existing one).
6. Drift gate (`tests/unit/scripts/selectors-catalog.test.ts`) AND format gate (`tests/unit/scripts/lessons-format.test.ts`) both run as part of `npm test` — they enforce the loop at PR-merge time.

### Test conventions

- `node:test` + `node:assert/strict`. No vitest, no jest.
- Mirror source layout: `src/foo/bar.ts` → `tests/unit/foo/bar.test.ts`.
- Imports use `.js` extension (ESM): `import { x } from "../../../src/foo/bar.js"`
- Frontend React components are NOT unit-tested. Verification is `npm run build:dashboard` + manual smoke.

### Commit conventions

- Conventional commits prefixes: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`.
- HEREDOC commit messages with `Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>`.
- One logical change per commit.
- Stage files by name. NO `git add -A`, NO `--amend`, NO `--no-verify`.
